# 12. Agent Catalog — Анализ реальных кейсов

## Цель документа

Разбор конкретных агентов, которые нужны пользователю. Для каждого: поведение, capabilities, реализация в текущей системе, что не хватает. Даёт основу для понимания gaps между текущей архитектурой и реальными потребностями.

**Статус:** Черновик. Каждый агент будет обсуждён отдельно.

---

## 1. Заметки (Notes)

### Оригинальное описание
> Сохраняет в памяти, потом может искать по векторам. Усиленный: умеет проставлять теги, знает имеющиеся теги в системе, может доставать по тегам, может строить связи, может доставать по цепочкам тегов.

### Наблюдения
- **Базовая версия (L2):** `agent_knowledge` уже поддерживает store + semantic search + tags. Это самый готовый к реализации агент.
- **Усиленная версия:** "знает имеющиеся теги" — нужен `list` с агрегацией по тегам (в agent_knowledge есть `list` с tag-фильтром, но нет "дай все уникальные теги"). "Строить связи" и "цепочки тегов" — это граф-подобная навигация, выходит за рамки простого tag-фильтра. Вопрос: достаточно ли prompt-driven поведения (L2 сам строит связи через множественные search-запросы) или нужен custom tool (H2)?
- **Gap:** Нет операции "list all unique tags" в agent_knowledge. Добавить просто (`SELECT DISTINCT unnest(tags)`).

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень (базовый) | L2 |
| Уровень (усиленный) | L2, возможно H2 для граф-навигации |
| Tools | agent_knowledge |
| Proactive | Нет (reactive only) |
| Готовность системы | ~90% для базового |

## 2. Писательский (Writing Assistant)

### Оригинальное описание
> Помогает делать записи по произведениям. Помогает углубиться в персонажа, арку, конфликт, мир. Сохраняет записи, может выдать обратно по поиску. Усиленный: у меня есть набор практик для развития разных сторон произведения, агент помогает мне, используя эти практики, из моих ответов достаёт информацию, обогащает собственные записи. Супер мощь: для разных проектов разные работы записей в базу, проекты как отдельная сущность, к каждому проекту записи по героям в целом или отдельным героям, по миру, по сюжетным аркам, при этом агент может обновлять или заменять записи; если есть противоречия, сообщает.

### Наблюдения
- **Базовая версия (L2):** `agent_knowledge` для текстовых записей + semantic search. Промпт описывает структуру (персонаж, мир, арка). Теги для категоризации (`["персонаж:Иван", "мир", "арка:главная"]`).
- **Усиленная версия:** "Практики" — это по сути методологические промпт-шаблоны. Агент должен вести диалог по методике, извлекать инфо из ответов, обогащать записи. Это prompt-driven — L2 справится, если промпт хороший.
- **Супер мощь:** "Проекты как сущность" + "записи по героям/миру/аркам" — это структурированная иерархия. `agent_memory` (collections = projects, keys = entities) + `agent_knowledge` (semantic search по текстам). "Обнаружение противоречий" — это самое сложное: нужен search по existing записям + LLM-анализ на конфликты. Prompt-driven, но требует multi-step: save → search related → compare → alert.
- **Gap:** Оба tool'а (memory + knowledge) одновременно нужны одному агенту. Сейчас это поддерживается через `tools: ['agent_memory', 'agent_knowledge']`. Нет архитектурного блокера. Вопрос в надёжности prompt-driven обнаружения противоречий — если LLM забывает проверять, это триггер для H2.

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень (базовый) | L2 |
| Уровень (усиленный) | L2 (prompt-driven практики) |
| Уровень (супер мощь) | L2→H2 (если противоречия ненадёжны) |
| Tools | agent_memory + agent_knowledge |
| Proactive | Нет |
| Готовность системы | ~80% (нужна проверка dual-tool сценария) |

## 3. Идеи проектов (Project Ideas)

### Оригинальное описание
> Тоже самое про идеи проектов. У меня часто есть идеи проектов, которые мне надо не просто записывать, но и критиковать, помогать находить пути обхода проблем и блокеров, развивать для достижения ценности и MVP. Усиленный: агент может следить за повторяющимися, связанными, похожими идеями, помогать мне комбинировать их.

### Наблюдения
- **Базовая версия (L2):** Очень похоже на Notes, но с аналитическим промптом: "При записи идеи — критикуй, предложи блокеры и пути обхода, предложи MVP". `agent_knowledge` для хранения + search.
- **Усиленная версия:** "Следить за похожими" — при каждом `store` агент должен делать `search` по existing идеям и сравнивать. Это prompt-driven multi-step workflow. Ключ — промпт должен предписывать: "после сохранения новой идеи, найди 3 самых похожих и проверь пересечения".
- **Gap:** Нет архитектурного блокера. Но hook мог бы автоматизировать "after:agent_knowledge.store → search similar". Hooks уже реализованы, но hook вызывает другой tool, а не "тот же tool с другими параметрами + LLM-анализ результата". Для полноценного "найди и сравни" нужен LLM в цепочке, а hooks — детерминированные. Это значит — полагаемся на prompt.

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень | L2 |
| Tools | agent_knowledge |
| Proactive | Нет |
| Готовность системы | ~90% |

## 4. Сайд квесты (Side Quests)

### Оригинальное описание
> Присылает каждый день 3 простых сайд квеста: например, пройтись по знакомым местам без наушников, сфотографировать 5 разных текстур, посидеть молча 5 минут на лавочке в парке. Усиленный: сохраняет то, что было предложено и то, что было сделано.

### Наблюдения
- **Базовая версия (L1):** scheduler (manage_schedule) отправляет ежедневный trigger → агент генерирует 3 квеста → send_message. Чисто proactive, без memory.
- **Усиленная версия (L2):** `agent_memory` для хранения предложенных квестов (collection: `quests`, key: дата) и выполненных (collection: `completed`). Агент при генерации проверяет историю, чтобы не повторяться.
- **Gap:** Нужен scheduler (manage_schedule tool). Он упомянут как доступный через `tools[]`, но надо проверить реальную интеграцию: может ли scheduler trigger вызвать агента? Или scheduler шлёт webhook в систему, а система активирует агента?

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень (базовый) | L1 |
| Уровень (усиленный) | L2 |
| Tools | manage_schedule, (agent_memory для L2) |
| Proactive | Да — ежедневный триггер |
| Готовность системы | ~60% (зависит от scheduler интеграции) |

## 5. Облигации (Bonds List)

### Оригинальное описание
> Выдает список облигаций, делая запрос на мой внешний сервис. Возможно, вариант, когда внешний сервер переходит в idle режим, к нему прилетает запрос, сам сервер делает обновление данных, потом отправляет агенту веб-хук, что обновился и готов, а уже сам агент запрашивает данные от сервиса.

### Наблюдения
- **Простой вариант (L1):** `http_request` в endpoint mode. Endpoint pre-configured с URL + auth. По запросу пользователя агент делает GET, парсит результат, отвечает.
- **Webhook-вариант:** Сложнее. Внешний сервер → idle → запрос будит его → сервер обновляет данные → webhook в Persais → Persais активирует агента → агент запрашивает данные. Это требует: (1) incoming webhook endpoint в Persais, (2) маршрутизация webhook → конкретный агент, (3) агент делает http_request после пробуждения.
- **Gap:** Incoming webhooks — в системе нет механизма "внешний сервис вызывает агента". Scheduler может trigger'ить, но произвольный HTTP callback — нет. Это крупный gap.

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень (простой) | L1 |
| Уровень (webhook) | Требует инфраструктуру |
| Tools | http_request (endpoint mode) |
| Proactive | Webhook-driven (не scheduler) |
| Готовность системы | ~70% для простого, ~30% для webhook |

## 6. Доступность облигации (Bond Availability Check)

### Оригинальное описание
> Проверка доступности определенной облигации по прямому запросу. У меня есть список облигаций, доступных на бирже, но мой провайдер облигаций может ещё не разместить у себя одну из облигаций, которая мне интересна. Поэтому мне нужен агент, который каждый день делал бы запрос на определенный url и узнавал, залистил ли провайдер нужную облигацию. Установили, когда надо. Удалили по завершению.

### Наблюдения
- **Реализация (L1):** scheduler (daily) + `http_request` (GET на URL провайдера) + промпт для парсинга ответа. Если облигация найдена — send_message. Если нет — молчит или "пока нет".
- **Паттерн:** Временный одноцелевой агент. Install → работает → цель достигнута → uninstall. Это интересный lifecycle pattern: агент с конечным временем жизни.
- **Gap:** Нет механизма "self-uninstall" или "complete task and notify". Агент не может сам себя удалить. Пользователь должен вручную сказать "удали". Или Master может предложить: "Облигация найдена, удалить агента?"

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень | L1 |
| Tools | manage_schedule, http_request |
| Proactive | Да — ежедневный scheduler |
| Lifecycle | Временный (install → goal → uninstall) |
| Готовность системы | ~60% (scheduler интеграция) |

## 7. Проверка курса (Exchange Rate Monitor)

### Оригинальное описание
> У меня есть сервис для таймеров кронос. Агент выставляет его, чтоб тот делал запросы в другой мой внешний сервис, чтоб тот делал запросы на биржу bybit и получал оттуда актуальный курс для обмена. В случае достижения определенных условий раньше сервис отправлял сообщения в собственного телеграм бота, но теперь может отправлять в агента и он будет сообщать мне. Усиленный: этот же сервис имеет api, чтоб можно было не дожидаться таймера, а уже прямо сейчас узнать актуальный курс. Тогда агенту надо будет делать запрос в этот сервис по url и сообщать мне результат.

### Наблюдения
- **Архитектура:** Cronos (внешний таймер) → HTTP запрос в rate-сервис → rate-сервис запрашивает bybit → при условии → webhook в Persais → агент сообщает. Плюс on-demand: пользователь спрашивает → агент делает http_request → отвечает.
- **Reactive часть (L1):** `http_request` endpoint mode → rate-сервис API → парсинг → ответ. Тривиально.
- **Proactive часть:** Снова incoming webhooks. Cronos/rate-сервис должен уметь дёргать Persais. Тот же gap что в "Облигации".
- **Ключевое наблюдение из оригинала:** "Вся логика там очень простая... без интеллектуальной обработки просто сообщает в чат." — Это не агент в полном смысле. Это алгоритмический обработчик. См. секцию "Mechanic: не только агенты" ниже.

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень (reactive) | L1 |
| Уровень (proactive) | Требует incoming webhooks |
| Tools | http_request, (manage_schedule) |
| Proactive | Webhook-driven от Cronos |
| Готовность системы | ~40% (webhook gap + вопрос: нужен ли LLM?) |

## 8. Состояние здоровья (Health Tracker)

### Оригинальное описание
> Очень хотелось завести агента, чтоб он помогал мне заполнять форму по здоровью. Я делаю аудио запись или текстовую, в которой раз-два в день делаю описание моего здоровья с учётом определенных моментов, но в целом в свободной форме. Агент записывает эти данные в базу. Уровень 1: просто выжимает мой текст, делает чистым, записывает в базу. Уровень 2: заполняет форму с определенными полями. Важна точность, так как я хочу потом взять все эти данные и на их основе попробовать обучать какую-то модель, чтоб потом делать предсказания о своём же состоянии на следующий день. Хотя бы попробовать найти закономерности.

### Наблюдения
- **Уровень 1 (L2):** Пользователь говорит/пишет → LLM извлекает суть → `agent_memory` (collection: `health`, key: дата). Голосовые сообщения уже поддерживаются через Deepgram (speech-to-text). Промпт: "Из свободного описания извлеки ключевые жалобы, симптомы, самочувствие. Запиши структурированно."
- **Уровень 2 (L2→H2):** "Форма с определенными полями" — LLM заполняет JSON-схему. Промпт описывает поля: `{energy: 1-10, mood: 1-10, sleep_hours: number, symptoms: string[], ...}`. L2 может — но "важна точность" это red flag для prompt-driven. LLM может пропустить поле, неправильно оценить шкалу. Если данные потом идут в ML — нужна детерминированная валидация (H2).
- **ML-pipeline:** Данные в `agent_memory` хранятся как JSONB — извлечь для обучения можно напрямую через SQL. Но нужна консистентная схема, а L2 не гарантирует это.
- **Gap:** Нет валидации формата записи. L2 может записать `{energy: "высокая"}` вместо `{energy: 8}`. Для ML это критично. Варианты: (1) Zod-валидация в custom tool (H2), (2) hook с валидацией after:agent_memory.save, (3) принять риск и чистить данные потом.

### Предварительная оценка
| Аспект | Значение |
|--------|----------|
| Уровень 1 | L2 |
| Уровень 2 | H2 (если нужна точность для ML) |
| Tools | agent_memory, (voice: уже есть через Deepgram) |
| Proactive | Нет (или reminder через scheduler) |
| Готовность системы | ~70% для L2, ~20% для H2 |

## Выявленные gaps

1. **Incoming webhooks** — внешние сервисы не могут вызвать агента (облигации, курс). Крупный gap.
2. **Scheduler↔Agent интеграция** — может ли scheduler trigger активировать конкретного агента? Нужна проверка.
3. **"List all tags"** в agent_knowledge — мелкий gap, легко добавить.
4. **Валидация формата данных** — L2 не гарантирует схему записей (здоровье, ML).
5. **Временные агенты** — нет lifecycle "install → goal reached → auto-uninstall" (доступность облигации).
6. **Dual-tool сценарий** — agent_memory + agent_knowledge одновременно (писательский). Нужна проверка.

## Отдельная идея: Mechanic — не только агенты

### Оригинальное описание
> Возможно, надо разделить этот сервис на более и менее интеллектуальные действия. Сейчас механик создает только агентов, но вообще он же мог бы создавать любые обработчики. И просто ручки для обработки данных тоже. Например, ту же проверку курса. Вся логика там очень простая. Внешний кронос делает запросы, системный обработчик просто делает запрос и записи в базу. И там же без интеллектуальной обработки просто сообщает в чат. Это значит, что механик должен уметь делать не только интеллектуальных полноценных агентов, но и просто логику для хуков и других алгоритмических простых тулов.

### Анализ

Это важная архитектурная идея. Суть: не всё, что делает система, требует LLM. Некоторые задачи — чисто алгоритмические:

- **Проверка курса:** webhook → fetch API → compare threshold → send message. Ноль LLM.
- **Облигации (webhook):** incoming request → fetch data → format → send. Ноль LLM.
- **Health validation:** after:agent_memory.save → validate JSON schema → reject if invalid. Ноль LLM.

Сейчас у нас два вида "единиц работы":
1. **Агенты** — LLM-driven, agentic loop, промпт + tools
2. **Hooks** — детерминированные, tool → tool, без LLM

Но hooks привязаны к агентам. А что если нужен "обработчик" который не принадлежит ни одному агенту? Или который вообще не использует LLM?

**Возможные направления:**
- **Расширить hooks** — разрешить system-level hooks (не привязанные к агенту)
- **Новая сущность "handler"** — алгоритмический обработчик, создаваемый Mechanic'ом
- **"Light tool"** — Mechanic создаёт не агента, а tool с простой логикой (JS-функция)

Это перекликается с идеей incoming webhooks: если система умеет принимать внешние вызовы и маршрутизировать их на handlers (не агенты), это покрывает курс, облигации, и другие алгоритмические кейсы.

**Статус:** Идея зафиксирована. Требует отдельного обсуждения и проектирования.

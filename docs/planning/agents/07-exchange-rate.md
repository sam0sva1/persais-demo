# Проверка курса (Exchange Rate Monitor)

## Оригинальное описание

> У меня есть сервис для таймеров кронос. Агент выставляет его, чтоб тот делал запросы в другой мой внешний сервис, чтоб тот делал запросы на биржу bybit и получал оттуда актуальный курс для обмена. В случае достижения определенных условий раньше сервис отправлял сообщения в собственного телеграм бота, но теперь может отправлять в агента и он будет сообщать мне. Усиленный: этот же сервис имеет api, чтоб можно было не дожидаться таймера, а уже прямо сейчас узнать актуальный курс. Тогда агенту надо будет делать запрос в этот сервис по url и сообщать мне результат.

## Анализ

### Текущая архитектура

```
Cronos timer → HTTP запрос в rate-сервис → rate-сервис → bybit API
  → при пороге → раньше: Telegram bot, теперь: Persais
```

### Reactive часть (L1)

Пользователь спрашивает "какой курс?" → агент делает `http_request` к rate-сервису API → парсит → отвечает. Тривиально.

### Proactive часть — две модели

**Модель A: Cronos → rate-сервис → Persais**
Cronos дёргает rate-сервис, rate-сервис при пороге шлёт webhook в Persais. Это требует generic incoming webhook (которого пока нет).

**Модель B: Cronos → Persais → rate-сервис**
Cronos дёргает Persais через scheduler webhook → агент делает `http_request` к rate-сервису → сравнивает с порогом → сообщает если нужно.

Модель B **уже реализуема**: `manage_schedule` (cron) + `http_request` (GET). Агент сам решает, уведомлять ли.

### Нужен ли тут LLM?

Из оригинала: "Без интеллектуальной обработки просто сообщает в чат." Это кандидат на алгоритмический handler (см. [mechanic-handlers.md](mechanic-handlers.md)). Но L1 агент с простым промптом тоже справится, пока handlers не реализованы.

## Что есть в системе

| Компонент | Статус |
|-----------|--------|
| `manage_schedule` tool | Готов |
| `http_request` tool | Готов |
| Cronos → Persais → agent pipeline | Готов |
| `manage_schedule` в CAPABILITY_TOOLS | Готов |
| Generic incoming webhooks | Нет — нужно для модели A |

## Что нужно для реализации (Модель B)

1. Настроить endpoint: URL rate-сервиса API
3. Написать промпт (запроси курс, сравни с порогом, сообщи если нужно)
4. `manage_agents.install(manifest)` + `manage_schedule.create(cron)`

## Предварительная оценка

| Аспект | Значение |
|--------|----------|
| Уровень | L1 |
| Tools | `manage_schedule`, `http_request` |
| Proactive | Да — cron trigger (модель B) или webhook (модель A) |
| Готовность системы | ~95% для модели B |
| Сложность реализации | Низкая |
| Альтернатива | Алгоритмический handler (без LLM) — см. mechanic-handlers.md |
